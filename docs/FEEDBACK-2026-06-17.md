## ✅ 已修

### (3) 聊天底色浓度默认应为 80%，实际却完全透明
- **根因**：`apps/web/src/lib/visual.ts` 的 `num()` 读取 localStorage 时
  `Number(get(k))`，而 `Number(null) === 0`（键未设置时 `get` 返回 `null`）。
  `0` 落在 `0..100` 区间检查内，于是**未设置的偏好被当成 0**，而不是回退到默认值
  `VISUAL_DEFAULTS.veilOpacity = 80`。CSS `:root` 的默认 `.8` 是对的，是 JS 用 0
  覆盖了它。同类问题也影响 `spriteOpacity`。
- **处置**：`num()` 增加 null/空串保护——未设置即返回默认值，再做数值与区间校验。
  同时修正了把这个 bug 当成预期行为写死的测试（`visual.test.ts`）。
- **验证**：147 个前端测试通过。

### (4) 一些应处理的 npm 库（deprecated + 1 个 high 漏洞）
- **根因**：`inflight / rimraf@2 / glob@7 / boolean` 这几个 deprecated 传递依赖
  **其实已经不在依赖树里**了（`npm ls` 为空）。真正残留的是一个 **high**
  级别的 esbuild/vite 漏洞（GHSA-gv7w-rqvm-qjhr：Deno 模块缺少二进制完整性校验）。
- **处置**：`vite` 7→8、`@vitejs/plugin-react` 5→6（vite 8 用 rolldown 打包，
  把 chunk 配置从 rollup 的 `manualChunks` 对象式迁移到 rolldown 原生
  `codeSplitting.groups`）。
- **验证**：`npm audit` → **0 vulnerabilities**；`npm run build` 干净；全部测试通过。

### (9) 删除「给用户选项」的那个工具（clarify）
- **根因**：`clarify` 工具让模型可以反过来弹选项给用户，容易让它走捷径而不是自己决断。
- **处置**：删除工具注册 `tools/builtin/clarify.py`，并从 `delegate_task` 的黑名单、
  `toolpacks/sandbox.json`、相关测试中移除。验证模型侧已**完全无法再调用** clarify。
- **重要说明（留待跟进）**：clarify 是一条**跨层**特性，除工具本体外还有一整套
  「交互式提问」管线：`protocol/codec` 的 `clarify_ask`/`clarify_reply` 事件、
  `server/dispatch.py` 的往返、`front/terminal.py` 的 stdin hook、以及 **7 个 React
  文件**的渲染。这套管线与 `permission_hook` 同构，是通用基础设施。工具删除后它**休眠
  且永不被触发**。彻底剔除需走「protocol 先行 + owner 签字」的协调改动（涉及宪法层
  codec 与整个 React 客户端），故本轮**只断源头（工具）**，管线作为 follow-up。

## 🔬 仅调研（owner 指示先搜方案、不要直接动手）

### (1) write_file 似乎有写入上限，约 12KB 处被截断
- **根因**：**不是工具的限制**。`tools/builtin/file_tools.py` 的 `write_file`
  没有大小上限（`_DEFAULT_MAX_READ_CHARS=100_000` 只管*读*）。真正的截断点是
  **模型的补全 token 上限**：`core/llm.py:_max_tokens_param()` 把
  `cfg.max_tokens`（`config.py` 默认 `LLM_MAX_TOKENS=4096`）作为 completion 上限，
  而工具调用的**参数(文件内容)也算在这条补全里**。约 4096 token ≈ 12KB 文本，模型在
  生成 `write_file` 参数中途撞上限被切断，于是它只能把文件拆成 part2/part3。
- **业界做法（调研结论）**：
  - 这正是 agent loop 里 `stop_reason == "length"`（OpenAI 称 `max_tokens`）
    在 `tool_use` 参数中途触发的「静默截断」，是已知坑（hermes #26425、openclaw #63210
    都在处理「检测并从输出截断中恢复」）。
  - 推荐组合拳：① **调高 `LLM_MAX_TOKENS` 默认值**（给写大文件留足空间）；
    ② **每次响应检查 `stop_reason`/`finish_reason`**，是 `length` 就显式报错而非
    把残缺参数喂给工具（符合本项目「No failure fallbacks—失败要可见」原则）；
    ③ 引导大文件用**分块 append / patch** 而非单次巨写。
- **待决策**：默认值调到多少、是否加截断检测。建议纳入重构 / owner 拍板。
- 参考：[hermes #26425](https://github.com/NousResearch/hermes-agent/issues/26425)、
  [openclaw #63210](https://github.com/openclaw/openclaw/issues/63210)、
  [Vellum: max-tokens](https://www.vellum.ai/llm-parameters/max-tokens)

### (8) 两个 web 工具都坏了，返回为空
- **根因（修正了之前的错误猜测）**：**不是缺 User-Agent**——`web.py:_http_get`
  确实设置了 `User-Agent`。真正原因是 `_search_duckduckgo()` 抓取
  `html.duckduckgo.com/html/`，而 DDG 在检测到自动化请求时返回 **403 / 空结果**
  （bot 检测 / 限流 / CAPTCHA），HTML 版面也可能已变动导致解析落空。
- **业界做法（调研结论）**：
  - DDG html 端点对脚本极不稳定；常见缓解：换 `lite.duckduckgo.com/lite`
    端点、加请求间隔与轮换代理，或干脆**改用稳定的搜索 API**（SerpAPI / ScrapingBee /
    Brave Search API / 开源 duckduckgo-scraper 等）。
  - 与 messaging 同属「需要真实联调的一轮修复」（roadmap C.3 已留此预算）。
- **待决策**：是「换端点 + 重试」轻修，还是接一个搜索 API 后端（`_resolve_search_backend()`
  已经是可插拔的）。建议纳入下一轮。
- 参考：[scrape DuckDuckGo SERP](https://brightdata.com/blog/web-data/how-to-scrape-duckduckgo)、
  [omkarcloud/duckduckgo-scraper](https://github.com/omkarcloud/duckduckgo-scraper)

## ⏸ 搁置

### 架构问题：用户进入/离开 session 的语句是否有必要、删掉能否让架构清晰很多？
- **本人结论（已与 owner 同步）**：可以删掉的是**注入进历史的 `[operator entered]`/
  `[operator left]` 标记句**、`presence.marker_text` 机制、以及 `on_attach`/`on_detach`
  改写 hook——因为 presence **本就**作为 `user_present` 事实挂在 volatile tail 里，这些
  注入句既冗余、又会**永久污染 transcript / 吃 context**。删掉=presence 变成纯粹「每轮重算
  的事实」，不再是写进历史的事件。**但**驱动「对话 ↔ 自我工作」模式切换的 attach/detach
  **状态机**是承重的，必须保留——删标记句不碰它。净简化「有意义」，但不会让 presence 模块
  整体坍缩。唯一代价：失去历史里「此刻刚进/刚出」的瞬时信号（时间戳/gap 已能近似表达）。
- **处置**：owner 今日决定**搁置**，不在本轮改动。结论留档备查。
